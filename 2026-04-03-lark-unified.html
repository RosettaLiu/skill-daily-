这个 Skill 是什么？

一句话：让 AI 助手直接操作飞书的一切 — 发消息、写文档、查日程、建任务、管表格，一行命令搞定。

这不是一个简单的"聊天机器人"。Lark Unified 是飞书/Lark 全平台的CLI 统一接入层，封装了 200+ 条命令覆盖即时通讯、云文档、电子表格、多维表格、日历、邮件、任务、Wiki、网盘、视频会议、通讯录共 11 个业务域。还附带 19 个 AI Agent 优化的子 Skill，以及自定义 Skill 创建能力。简单说：你不用再打开飞书客户端了，AI 帮你干完。

核心理念：把飞书变成 AI 的"手"

🧠 为什么需要一个飞书 CLI？

飞书是国内企业协作的主流平台，但日常操作都绑在 GUI 上——切群、找文档、查日程、建任务，每天重复几十次。

Lark Unified 的设计逻辑很清晰：把所有飞书操作都变成命令行指令，让 AI 能直接执行。

  • 统一入口：不管是发消息还是建表格，都是 lark-cli <domain> <resource> <method>
  • 快捷方式优先：高频操作有 +shortcut 封装，一行搞定
  • 双身份模式：--as user（以你的身份操作）或 --as bot（以应用机器人身份操作）
  • Agent 友好：所有输出支持 JSON/CSV/YAML/Table 格式，方便程序解析

🔑 双身份认证机制（关键设计）

同一条命令，用不同身份执行可能结果完全不同。这是整个 Skill 最核心的概念：

👤 User 身份

使用 --as user
Token: user_access_token
权限 = 你自己能做什么
适合：个人操作、查自己的日程消息

🤖 Bot 身份

使用 --as bot
Token: tenant_access_token
权限 = 应用的 OAuth 范围
适合：建群、群发消息、管理操作

⚠️ 实际使用中最常见的坑：权限不足返回空结果或报错。永远先检查身份 + 权限范围。

1
即时通讯：一行命令发消息建群
▼

飞书消息支持 text/post/image/file/audio/video/card 等类型。Bot 身份可以群发，User 身份可以搜自己的聊天记录。

# 搜索群聊 lark-cli im +chat-search --keyword "产品组" # 发消息到群 lark-cli im +messages-send --chat-id oc_xxx --text "日报已提交" # 搜索消息记录（限定时间范围和发送者） lark-cli im +messages-search --query "deadline" --from-user ou_xxx --start-time 2026-04-01 --end-time 2026-04-03 # 下载消息中的图片/文件 lark-cli im +messages-resources-download --message-id om_xxx

⚠️ Bot 身份取消息时，发送者名字可能显示为 open_id 而非真名。需在开发者后台设置应用可见范围。

2
多维表格：AI 操作结构化数据的最佳入口
▼

多维表格（Base）是飞书最强大的数据管理工具。支持 20+ 字段类型，含公式、关联、汇总、自动化工作流。

# 列出某个 Base 的所有表 lark-cli base tables list --base-id app_xxx # 查字段定义（了解表结构） lark-cli base +fields-list --base-id app_xxx --table-id tbl_xxx # 查询记录（带过滤和排序） lark-cli base +tables-records-list --base-id app_xxx --table-id tbl_xxx --filter "[condition]" --sort "[sort_spec]" --limit 100 # 创建新记录 lark-cli base +tables-records-create --base-id app_xxx --table-id tbl_xxx --data '{"fld_name":"value"}' # 触发自动化工作流 lark-cli base workflows trigger --base-id app_xxx

支持的字段类型：文本、长文本、数字、单选/多选、日期、复选框、评分、用户、部门、查找引用、公式、汇总、附件等。

3
云文档 & Wiki：Wiki 链接的"坑"必须知道
▼

飞书文档有多种类型（docx/doc/sheet/bitable/slides/mindnote），Wiki 链接可能指向其中任何一种。不能假设 wiki_token 就是文档 token。

# ❌ 错误：直接用 wiki_token 当 file_token # ✅ 正确：先解析 Wiki 节点 lark-cli wiki spaces get_node --params '{"token":"wikcnxxxxxxx"}' # 返回 obj_type (docx/sheet/bitable...) 和 obj_token # 然后根据 obj_type 调用对应 API lark-cli doc +docs-get --document-id [obj_token] # if obj_type=docx lark-cli sheets +spreadsheets-read --spreadsheet-id [obj_token] # if obj_type=sheet

⚠️ 这是新手最容易踩的坑。Wiki 页面的 URL 中的 token 不是文档 token，必须先查询转换。

4
日历智能调度：查空闲 + 推荐时间
▼

不只是查日程，还能查多人空闲时段、自动推荐最佳会议时间。AI 安排会议的完美工具。

# 查看今日日程 lark-cli calendar +agenda # 查某人的空闲时段 lark-cli calendar +freebusy --user ou_xxx --start-time 2026-04-03T08:00:00 --end-time 2026-04-03T18:00:00 # 智能推荐会议时间（考虑多人日程） lark-cli calendar +suggestion --user ou_xxx --duration 60 --start-time 2026-04-03 --end-time 2026-04-07 --attendees "alice@co.com" "bob@co.com" # 创建会议并邀请参与者 lark-cli calendar +create --user ou_xxx --title "周会" --start-time 2026-04-04T14:00:00 --end-time 2026-04-04T15:00:00 --attendees "alice@co.com"
5
电子表格自动化：读写追加一气呵成
▼

飞书表格操作使用 A1 记法标注范围，支持批量读写、查找替换。自动化数据录入的利器。

# 创建表格 lark-cli sheets +spreadsheets-create --title "Q2 数据" # 写入表头 lark-cli sheets +spreadsheets-append --spreadsheet-id spr_xxx --range "Sheet1!A1" --values '[["姓名","部门","金额"]]' # 追加数据 lark-cli sheets +spreadsheets-append --spreadsheet-id spr_xxx --values '[["张三","产品",5000],["李四","技术",8000]]' # 读取数据 lark-cli sheets +spreadsheets-read --spreadsheet-id spr_xxx --range "Sheet1!A1:C100" # 查找 lark-cli sheets +spreadsheets-find --spreadsheet-id spr_xxx --find "张三"

7
安装配置：Device Flow 免 TTY 认证
▼

内置 lark_setup.py 脚本实现了 Device Flow 认证，无需交互式终端（TTY）就能完成配置。

# Step 1: 安装 lark-cli npm install -g @larksuite/cli # Step 2: 运行自动配置脚本（飞书版） python3 lark_setup.py # Lark 国际版 python3 lark_setup.py --brand lark # 无法打开浏览器时 python3 lark_setup.py --no-browser # 验证配置 lark-cli config view

脚本流程：请求 Device Code → 浏览器授权 → 轮询等待 → 保存配置。中间的 authorization_pending HTTP 400 是正常行为，不是错误。

🚫 绝对不要运行 lark-cli config init --new 或 lark-cli config init（交互式），会在 AI 环境中卡死！

8
进阶能力：事件订阅 + 自定义 Skill
▼

除了主动调用 API，还支持被动订阅实时事件（WebSocket），以及用 Skill Maker 创建自定义工作流。

# 订阅消息事件（实时推送） lark-cli event subscribe --event-type message.receive --endpoint wss://your-endpoint # 探索 API lark-cli openapi search --query "message" lark-cli openapi get --api im.messages.send # 白板/绘图（Mermaid 语法） lark-cli whiteboard create --title "架构图" --content "graph LR; A[Start] --> B[Process] --> C[End]"

支持的实时事件：消息创建/更新/删除、文档更新、多维表记录变更、日历事件变化等。

内置 2 个工作流模板：Meeting Summary Workflow（会议纪要聚合）和 Standup Report Workflow（站会日报生成）。

飞书重度用户必装，但门槛不低

👍 优点

  • 覆盖面碾压级。11 个业务域、200+ 命令、12 篇参考文档、19 个 Agent 子 Skill。在"飞书自动化"这个赛道上，这是我见过的最完整方案。别的 Skill 能操控一两个飞书功能，这个直接把整个飞书平台变成了 AI 的工具箱。
  • 快捷方式设计精良。+messages-send、+agenda、+freebusy 这些高频操作都有一步到位的快捷封装。不用背 API 结构，一行命令解决问题。高频路径做到了真正好用。
  • 双身份模型是正确的架构决策。User 身份和 Bot 身份的分离，让权限边界清晰。个人操作不会越界，应用自动化有独立权限空间。这是真正考虑过安全和实际部署场景的设计。
  • Device Flow 安装脚本解决了真痛点。在 AI 环境（无 TTY）中配置飞书凭据一直是个头疼问题。216 行的 lark_setup.py 优雅地解决了这个问题，连 Lark 国际版切换都考虑到了。
  • 文档架构合理。一个 SKILL.md 总览 + 12 篇领域参考文档的分层结构，既保证了 AI 能快速定位功能，又不会因为单文件过大导致 Token 浪费。

👎 缺点

  • 入门门槛高得吓人。安装 lark-cli → 创建飞书应用 → 配置 OAuth Scope → 发布应用版本 → 运行认证脚本 → 还要理解 User/Bot 身份差异。对非开发者来说，光配置就可能要折腾一两个小时。对比 Humanizer 的"装上就用"，这个 Skill 的上手成本是天花板级别。
  • 权限配置是黑洞。30+ 个 OAuth Scope，文档里说了"添加权限后必须发布新版本再重新获取 Token"，但实际操作中报错信息模糊。返回空结果也不会告诉你是权限问题还是查询条件问题。调试体验很差。
  • 强依赖飞书生态。如果你的公司用钉钉或企业微信，这个 Skill 完全无用。而且飞书 API 本身有版本更新、字段变更的风险，CLI 不一定能及时跟进。这是平台绑定的固有风险。
  • 缺少真实工作流示例。每个域的文档都有命令示例，但缺少跨域组合场景。比如"查今天日程 → 给缺席者发消息 → 更新任务状态 → 记录到表格"这种端到端自动化 demo 一个都没有。200+ 命令是零件，但没给蓝图。
  • scripts 目录只有配置脚本。和小红书 Skill（13个Shell脚本）或 Content Factory（8个 Prompt 模板）相比，这个 Skill 没有任何预制的业务自动化脚本。用户拿到的是"全能工具箱"而不是"拿来就用的方案"。

💡 谁最该用？

最佳人选 飞书重度用户 + 有技术背景（或有 AI 助手帮忙配置）——这个 Skill 能把你每天在飞书上的大量重复操作全部自动化。一旦配好，效率提升是指数级的。

进阶场景 需要飞书数据自动化的团队——多维表格 + 消息通知 + 任务管理的组合，能做出非常强大的内部工具流。

不建议 非飞书用户、纯内容创作者、技术恐惧症——配置成本太高，对你的工作流几乎没有帮助。老实用飞书客户端就好。

Buddy 的话：这是一个"天花板很高、地板也很高"的 Skill。200+ 命令的能力上限很恐怖，但大多数人可能连安装都完成不了。如果你的日常工作在飞书上，而且你（或你的 AI 助手）有能力搞定配置，那么这绝对值得投入时间。否则，先装个 Humanizer 解决眼前问题吧。
