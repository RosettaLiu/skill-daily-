📚 ArXiv Reader

LLM 驱动的论文分类与深度阅读 Agent 系统
AI 工具 学术研究 多 Agent 协作 LangChain v1.0.3
📦 41 个文件 🐍 Python + LangChain 🤖 3 个 Agent 角色 📂 8 个学科分类 ⚙️ 需配置 LLM API
🔗 ArXiv 官网 🦜 LangChain 文档
这个 Skill 是什么？

一句话：给一个 arXiv 论文 ID，自动分类 + 多轮深度阅读 + 生成结构化中文笔记。

这不是简单的"翻译摘要"工具。它是一个三 Agent 协作系统：Classifier 先判断论文属于哪个研究领域，Reader 用 Two-Pass（甚至三轮）深度阅读逐章分析 LaTeX 源码，Summary Agent 兜底处理未分类论文。8 个预设学科方向各有专属阅读指南，输出 Obsidian 友好的 Markdown 笔记。

核心理念：像研究员一样读论文，而不是像翻译机

为什么需要多轮阅读？

人类研究员读论文不是一遍过的。先看摘要和引言判断方向，再深入方法和实验细节，最后看附录补充。这个 Skill 完全模拟了这个过程：

  • Pass 1 — 宏观框架：读 Abstract + Introduction + Preliminaries + Contributions + Limitations，抓住论文核心，提出待深入的问题
  • Pass 2 — 方法细节：基于第一轮总结，读方法、实验、分析等主体内容，提取公式和关键结果，并决定是否需要读附录
  • Pass 3（可选）— 附录补充：只在 Reader Agent 明确输出 APPENDIX_NEEDED: YES 时触发，只输出增量信息

领域专属阅读，不是通用总结

每个学科分类都有独立的 reading_prompt.md，告诉 Reader 该关注什么。比如：

  • Agent Systems — 必须记录架构设计、工具使用方式、记忆管理、多Agent协作
  • LLM Training — 必须记录损失函数公式、优化器配置、数据来源和规模
  • Inference Accelerate — 必须记录加速比、延迟、显存占用、硬件环境
  • Technique Report — 98 行阅读指南，从预训练到RLHF到安全对齐全链路覆盖

这意味着同一篇论文被分到不同分类，会得到完全不同视角的阅读笔记。

1
ArXiv Fetcher — 双策略论文获取
▼

支持两种获取方式，有优雅的降级策略：

  • RSS 订阅（优先）— 快速获取当日新论文，按分类订阅 arXiv RSS 频道
  • arXiv API（降级）— RSS 无结果时调用官方 API，支持按日期和已知ID过滤

LaTeX 源码获取也有双保险：先尝试 arxiv_to_prompt 库（保留结构化信息），失败则直接下载 e-print tar/gz 包并自动查找 \begin{document} 所在的主 .tex 文件。

# 单篇论文模式 uv run python main.py --arxiv-id 2401.12345 uv run python main.py --arxiv-id https://arxiv.org/abs/2401.12345 # 指定分类阅读 uv run python main.py --arxiv-id xxxx --category rag_and_retrieval # 查看所有可用分类 uv run python main.py --list
2
Classifier Agent — 智能论文分类
▼

基于论文标题和摘要，输出 JSON 格式的分类结果：

{ "category": "rag_and_retrieval", "confidence": 0.87, "reasoning": "论文提出了新型检索增强生成架构..." }

设计亮点：

  • 温度设为 0.1，追求分类稳定性
  • 置信度低于 0.4 自动降级为 general，宁可漏分不错分
  • Robust JSON 解析：先尝试直接 parse，失败后正则提取 {...}，再失败返回 general
  • 分类描述从 _metadata.md 动态拼装，新增分类零代码
3
Reader Agent — 多轮深度阅读核心
▼

这是整个系统最有料的部分。三轮阅读流水线：

  • Pass 1（宏观框架）— 输入 Abstract + Introduction + Preliminaries 等（截断30K字符），输出初步总结和待深入问题
  • Pass 2（方法细节）— 输入第一轮总结 + 论文主体内容（截断50K字符），输出完整笔记并决定是否读附录
  • Pass 3（附录补充）— 仅在检测到 APPENDIX_NEEDED: YES 时触发，只输出增量信息

设计精巧之处：Pass 2 不是简单继续读，而是拿着 Pass 1 的问题去找答案，同时生成全新的完整笔记。附录判断也不靠规则——让 LLM 自己决定要不要继续读。

# reader_agent.py 关键逻辑 def read_paper(self, paper_info, parsed_paper): # Pass 1: 宏观框架 initial_summary = self.chain.invoke(first_pass_msg) # Pass 2: 方法细节 detailed_notes = self.chain.invoke(second_pass_msg) # Pass 3: 附录（可选） if self._should_read_appendix(detailed_notes) \ and parsed_paper.has_appendix: appendix_notes = self.chain.invoke(appendix_msg) detailed_notes += appendix_notes
4
LaTeX Parser — 论文结构化解析
▼

不是简单的文本提取，而是理解论文结构：

  • 提取 \begin{abstract}...\end{abstract}
  • 识别 \section{}、\subsection{}、\subsubsection{} 三级标题
  • 定位 \appendix 标记，区分正文和附录
  • 识别参考文献边界（\bibliography / \printbibliography），自动排除

关键设计：ParsedPaper dataclass 提供了 first_pass_text、main_body_text、appendix_text 三个属性，用关键词（introduction, background, contribution 等）自动将章节分配到对应阅读轮次。

轻量清理策略也很聪明：只移除 \label{}、\vspace{} 等噪音元素，保留大部分 LaTeX 命令——因为"LLM 读 LaTeX 没问题"。

5
Skill Loader — 零代码扩展分类
▼

插件式分类架构，添加新学科只需要两个文件：

skills/your_new_category/ ├── _metadata.md # 分类描述（给 Classifier 用） └── reading_prompt.md # 阅读指南（给 Reader 用）

Loader 自动扫描 skills/ 目录下的所有子文件夹（跳过 _ 开头），读取 metadata 和 reading_prompt。无需修改任何 Python 代码，重启即可识别。

这意味着你可以自定义任何研究领域——量子计算、生物信息、材料科学——只要能写出领域阅读指南，Agent 就能按你的要求去深读论文。

6
Prompt 工程 — 六套精心设计的模板
▼

6 个独立的 Prompt 文件，各有分工：

  • classifier_system.md — 分类专家身份，输出 JSON，含格式约束和置信度要求
  • reader_system.md — "资深 AI 研究员"身份，注入领域 reading_prompt，要求中文输出、保留 LaTeX 公式
  • reader_first_pass_user.md — 第一轮阅读指令：高屋建瓴，允许提出问题留给下一轮
  • reader_second_pass_user.md — 第二轮阅读指令：深入分析 + 整合第一轮 + 附录决策
  • reader_appendix_pass_user.md — 附录指令：只输出增量信息，有"附录无重要补充内容"的退出机制
  • summary_system.md — 快速总结身份，给 general 分类兜底

最妙的设计是 reader_first_pass 里的"你可以提出问题，交给后面的 agent 来解决"——这让第二轮阅读有了明确的目标驱动，不是漫无目的地重复读。

7
LLM 配置与缓存系统
▼

基于 LangChain 的 ChatOpenAI，支持 OpenAI 兼容 API（包括 DeepSeek 等）：

  • .env 文件配置 API 密钥、Base URL、模型名、温度、Max Tokens
  • LLM 实例缓存：相同参数配置只创建一次，避免重复初始化
  • 不同 Agent 使用不同参数：Classifier 温度 0.1（稳定），Reader 温度 0.3 + 16K tokens（详细），Summary 温度 0.2 + 4K tokens（简洁）
# .env 配置 LLM_API_KEY=sk-xxxxx LLM_BASE_URL=https://api.openai.com/v1 LLM_MODEL=gpt-4o LLM_TEMPERATURE=0.3 LLM_MAX_TOKENS=16000

架构教科书级，但入门门槛不低

👍 优点

  • 多轮阅读设计是真正的差异化。市面上 90% 的"论文阅读工具"就是把 Abstract 翻译成中文。这个 Skill 让 LLM 模拟研究员的阅读习惯——先扫全局、再挖细节、可选读附录。Pass 1 提问 + Pass 2 回答的接力设计非常精巧。
  • 领域专属阅读指南是杀手级特性。LLM Training 的阅读指南要求记录损失函数公式和优化器配置，Inference Accelerate 要求记录加速比和硬件环境——这不是通用 prompt 能做到的深度。Technique Report 的 98 行阅读指南简直是"如何读技术报告"的教学范本。
  • 插件式扩展架构。添加新学科分类不需要动一行 Python 代码，只写两个 Markdown 文件。这是正确的架构设计——把领域知识和代码逻辑完全解耦。
  • LaTeX 解析策略务实。不做"完美解析"，只做"够用的结构化"。保留大部分 LaTeX 命令让 LLM 自己理解，而不是试图转成纯文本。这个判断很对——LLM 确实能读懂 LaTeX。
  • 工程化程度高。LLM 缓存、优雅降级（RSS→API、LaTeX→摘要）、Robust JSON 解析——处处都有工程师的思考。

👎 缺点

  • 门槛高，小白劝退。需要 Python、uv、LLM API Key、理解 .env 配置、了解 arXiv ID 格式——这不是"装上就能用"的工具。对非技术用户来说，光是搞定 LangChain 依赖就够头疼了。
  • Token 消耗恐怖。一篇论文 3 轮阅读，每轮 16K tokens 输出 + 30-50K 输入，总消耗可能到 100K+ tokens。按 GPT-4o 价格算，读一篇论文可能要 0.5-1 美元。如果批量处理，钱包会哭。
  • 没有持久化存储。笔记生成后直接 print 到 stdout，没有数据库、没有文件输出、没有搜索功能。要自己想办法把输出存到 Obsidian 里。作为一个"论文阅读系统"，少了最关键的知识管理环节。
  • 分类器是硬伤。只有 8 个分类，覆盖面有限。搞 NLP 的论文可能被分到 general（因为没有 NLP 分类），然后只能用摘要做快速总结——浪费了深度阅读的能力。
  • RSS 获取策略在单篇模式下无用。fetch_papers() 方法支持批量获取，但 main.py 只暴露了单篇接口。批量处理的基础设施建好了，但入口没开放。

💡 谁最该用？

强烈推荐 AI 领域研究生和研究员 — 你已经有读论文的习惯，这个工具能帮你把每篇论文的阅读笔记标准化和结构化。8 个预设分类恰好覆盖 AI 核心方向。

推荐 技术负责人 / 算法工程师 — 用来快速扫描新论文，判断哪些值得精读。Classifier 的分类结果本身就有筛选价值。

谨慎 非 AI 领域研究者 — 你需要自己写 _metadata.md 和 reading_prompt.md 来适配你的学科。框架支持，但开箱即用仅限 AI。

底线建议：就算不跑代码，这个 Skill 的 8 套 reading_prompt.md 单独拿出来就是极好的"如何读论文"教程。特别是 LLM Training 和 Technique Report 的阅读指南，值得每个做 AI 的人收藏。

--- name: arxiv-reader description: 利用python，指定某个arxiv_id/url， 基于 LLM Agent 对这篇arxiv论文进行分类与深度阅读，直接print打印阅读笔记 description_zh: "基于 LLM 的 ArXiv 论文分类与深度阅读工具" description_en: "LLM-based ArXiv paper classification and in-depth reading tool" version: 1.0.3 metadata: openclaw: requires: bins: - uv env: - LLM_API_KEY - LLM_BASE_URL - LLM_TEMPERATURE - LLM_MAX_TOKENS primaryEnv: LLM_API_KEY --- ## 快速开始 ### 1. 配置 `.env` ```bash cp .env.example .env # 或直接编辑 .env ``` 确定你已经配置了： - `LLM_API_KEY` — OpenAI 或兼容 API 的密钥 - `LLM_BASE_URL` — API 地址 ### 2. 运行 ```bash uv venv uv pip install -r "{baseDir}/requirements.txt" # 单篇论文模式：指定 arxiv_id 或 URL uv run python "{baseDir}/main.py" --arxiv-id 2401.12345 uv run python "{baseDir}/main.py" --arxiv-id https://arxiv.org/abs/2401.12345 uv run python "{baseDir}/main.py" --arxiv-id https://arxiv.org/pdf/2401.12345.pdf # 指定以特定类别阅读 uv run python "{baseDir}/main.py" --arxiv-id xxxx --category yyy # 查看所有类别 uv run python "{baseDir}/main.py" --list ``` ## 添加新的阅读分类 在 `skills/` 下新建文件夹，包含两个文件： ``` skills/your_new_category/ ├── _metadata.md # 分类描述（告诉 Classifier 什么论文属于这个类别） └── reading_prompt.md # 阅读指南（告诉 Reader Agent 重点关注什么） ``` 重启即可自动识别，无需修改任何代码。 ## Python包 - LangChain 1.x — Agent 框架（基于 LangGraph） - LangChain OpenAI — LLM 接口（兼容 DeepSeek 等 OpenAI-compatible API） - arxiv — 官方 Python 库 - arxiv-to-prompt 获取arxiv论文latex源码
